# Port plan: unity-procedural-dungeons -> amaze

## Goal
Port the Unity procedural dungeon generation concepts from `sunsided/unity-procedural-dungeons` into this Rust workspace, extend local UIs to visualize dungeon-specific outcomes, and keep maze/path solving behavior consistent and verifiable.

## Source analysis (GitHub MCP)

### Unity features to port (core)
- `Assets/Scripts/DungeonManager.cs`
  - Dungeon modes: `Caverns`, `Rooms`, `Winding` (`DungeonType.cs`).
  - Floor generation via random walk variants:
    - Caverns: unconstrained random walk.
    - Rooms: long constrained walks + room stamping.
    - Winding: same as Rooms, with probabilistic room suppression.
  - Post-processing semantics:
    - Wall placement around floor tiles.
    - Exit location at last walker position.
    - Optional rounded-edge bitmask classification.
- `Assets/Scripts/Enemy.cs` + `Node.cs`
  - Grid-chasing with A*-like Manhattan heuristic and orthogonal movement.
- `Assets/Scripts/TileSpawner.cs`
  - Expands floor tiles and places walls in empty neighboring cells.

### Current Rust baseline (local repo)
- `amaze` already has multiple perfect-maze generators, `Wall4Grid`, and solvers (`BFS`, `DFS`, `A*`, dead-end filling).
- `amaze-gui` already supports:
  - Algorithm selection,
  - Generation-step animation,
  - Start/end selection,
  - Path highlighting (currently BFS).
- `amaze-cli` currently generates and renders mazes only.

## Design approach

### 1) Introduce dungeon-specific representation, keep maze representation intact
- Add a dungeon-centric grid representation in `amaze` (e.g., `DungeonGrid`):
  - Tile type enum (`Empty`, `Floor`, `Wall`, optional metadata flags for edge-mask/exit/item/enemy markers).
  - Efficient floor membership (hash set or bit grid) to avoid O(n) contains checks present in Unity code.
- Keep `Wall4Grid` untouched for existing maze algorithms.
- Add conversion layer:
  - `DungeonGrid -> PassabilityGrid` (or equivalent adjacency) for solver reuse.
  - Optional `DungeonGrid -> Wall4Grid` where useful for unified rendering path.

### 2) Add Unity-parity dungeon generators under `amaze::generators`
- Implement:
  - `DungeonCavernsWalk`
  - `DungeonRoomsWalk`
  - `DungeonWindingWalk`
- Preserve behavior knobs:
  - `total_floor_count`
  - long-walk length range
  - room half-width/half-height ranges
  - winding hall probability
  - seed-driven determinism
- Emit step events compatible with existing animation UX (extend `GenerationStep` or add `DungeonGenerationStep`).

### 3) Wall and edge classification post-process
- Implement wall placement around floor cells in one pass over floor bounds + margin.
- Implement optional 4-bit edge mask classification matching Unity’s top/right/bottom/left occupancy logic.
- Track exit tile = last accepted walker floor tile.

### 4) Solver compatibility and parity checks
- Reuse existing `BfsSolver` / `AStarSolver` on converted passability representation.
- Add dungeon-focused solver tests:
  - Reachability from spawn to exit.
  - BFS shortest-path sanity.
  - A* path length equals BFS path length for unweighted orthogonal movement.
  - Determinism under fixed seeds.
- Add regression fixtures for a small set of seeds and parameters.

### 5) UI extensions (amaze-gui)
- Add a “Mode” switch: `Maze` vs `Dungeon`.
- For `Dungeon` mode:
  - Dungeon type selector: `Caverns`, `Rooms`, `Winding`.
  - Controls for floor count and winding probability.
  - Render tile classes with distinct colors (floor/wall/empty), plus exit marker.
  - Animate dungeon generation steps similarly to maze generation animation.
  - Keep start/end click-to-solve workflow and display BFS/A* path overlays.
- Preserve existing maze UI behavior unchanged.

### 6) CLI extensions (amaze-cli)
- Add `gen-dungeon` subcommand with arguments:
  - `--type` (`caverns|rooms|winding`)
  - `--seed`, `--floor-count`, `--winding-probability`, bounds/ranges as needed
  - optional output style for ASCII/image-compatible dungeon rendering
- Ensure old `gen` command remains backward compatible.

## Implementation phases

### Phase A: Core model + generation
1. Add dungeon data structures and helpers in `amaze/src`.
2. Implement three dungeon generators with deterministic RNG.
3. Add post-processing (wall placement, exit tile, edge mask metadata).
4. Add conversion to solver-compatible graph/passability representation.

### Phase B: Tests and parity validation
1. Unit tests for each generator invariants.
2. Seed determinism tests.
3. Solver equivalence tests (BFS vs A* lengths).
4. Integration tests on representative parameter sets.

### Phase C: GUI integration
1. Introduce mode selection and dungeon control panel.
2. Add dungeon rendering layer + overlays (exit, path, hover/select).
3. Wire animation timeline for dungeon step events.
4. Keep pan/zoom and reset UX consistent.

### Phase D: CLI and docs
1. Add `gen-dungeon` CLI path.
2. Update `README.md` and `ALGORITHMS.md` with dungeon generation + solver notes.
3. Provide usage examples for GUI and CLI dungeon modes.

### Phase E: Validation
1. `cargo test --workspace`.
2. Run GUI manually for all dungeon types + a few seeds.
3. Verify no regressions in existing maze generation and solvers.

## Compatibility and risk notes
- Existing maze APIs should remain stable; dungeon support should be additive.
- Representation split avoids forcing maze code into non-perfect-maze semantics.
- Main risk: overfitting Unity quirks that are engine-specific (physics/colliders). Mitigation: replicate algorithmic intent (grid occupancy + Manhattan solving), not Unity object lifecycle.
- Main UX risk: cluttered GUI controls. Mitigation: mode-gated panels and sensible defaults.

## Defaults to apply during implementation
- Solver overlay in dungeon mode will default to BFS (with optional A* toggle if low effort).
- Scope locked: generation + visualization + path solving parity only (selected), excluding enemy AI loop and gameplay systems.
- Keep deterministic RNG behavior by threading seed through all dungeon generation steps.

## Deliverables
- New dungeon generation module(s) in `amaze`.
- Tests covering generator invariants and solver parity.
- Extended `amaze-gui` visualizing dungeon generation and solve paths.
- Extended `amaze-cli` command(s) for dungeon generation.
- Updated docs describing algorithms and usage.
